# 教学行动指南 (Action Guide) - V2.0

本指南用于统一教学理念、内容结构与落地流程，适用于两条主线的全部课程建设与复盘。

## 1. 教学定位 (Philosophy)
- **受众**：希望以正统、严肃方式掌握 AI 时代编程能力的初学者。
- **基调**：**专业、严谨且亲切 (Professional, Rigorous & Welcoming)**。
    - **隐喻原则**：使用贴近生活的隐喻（如“微信”、“工作台”），帮助理解。隐喻应自然引入，避免突兀或过于复杂的术语。
    - **初学者视角 (Beginner Mindset)**：
        - **打破“知识诅咒”**：不要机械地假设用户懂“术语”，任何新概念首次出现时都应给出通俗易懂的解释。
        - **一致性与回顾 (Consistency & Recall)**：在引入新概念时，显式回顾前序知识点，建立连贯的知识链条。
- **目标**：不只是“跑通代码”，而是建立对开发环境与 AI 工具的清晰认知，培养良好的开发习惯。
    - **一致性与回顾 (Consistency & Recall)**：
        - **定稿不可变 (Immutable Foundation)**：已定稿课程（目前为 M1L1, M1L2, M1L3, M1L4, M1L5, M1L6）被视为只读基座。严禁修改其定义好的課程順序或核心概念。任何新內容必須適配既定路徑，而不能要求前文配合修改。
        - 新修改的内容必须与已定稿课程的用词严格保持一致。
        - 在引入新概念时，必须显式回顾引用的前序知识点（如 M1L3 需回顾 M1L2 的 Path 概念），建立知识链条。
### 1.1 避坑指南：对抗“知识诅咒”的抽象原则
在编写任何新课件时，必须对照以下三种典型认知错误进行自查：

1.  **拒绝“术语孤岛” (The Concept Jump)**：
    *   **原则**：严禁在没有定义或隐喻铺垫的情况下，直接使用技术专有名词。
    *   **典型反例**：在 L5 中直接使用“解释器 (Interpreter)”。虽然前序课程一直在运行 Python，但“解释器”这个角色从未登场，这属于认知突跳。
    *   **要求**：首次出现新概念必须配套“降维打击”式的定义（如：翻译官/管家）。

2.  **拒绝“叫法漂移” (Label Inconsistency)**：
    *   **原则**：严禁对同一个行为在不同阶段使用不同的非铺垫名称，必须维持“知识锚点”的稳定性。
    *   **典型反例**：L3 称 `a = 10` 为“记忆 (Memory)”，L5 若突然改称“变量赋值 (Assignment)”，会割裂学员的直觉。
    *   **要求**：沿用已有描述，并在引入专业词汇时明确告知这是“旧知识的新名字”。

3.  **拒绝“隐性经验假设” (Implicit Assumption)**：
    *   **原则**：不要因为“这些词太常见了”就默认学员懂。
    *   **典型反例**：默认学员理解“后缀名 (Extension)”对操作系统的意义。
    *   **要求**：任何作为教学关键支点的操作，其背后的物理规则（如文件关联）必须进行极简勾勒。

## 2. 课程体系 (Curriculum Tracks)
### Learn Python with AI IDE (基础课)
- **架构**：**Module 制**（非 Week 制）。
- **内容**：分两大阶段。
    - **Module 1**: 从“原声终端”到“AI IDE 工作流”。建立工程基座。
    - **Module 2**: 在 AI IDE 辅助下系统学习 Python 核心体系（对齐 Ng 课程）。
- **目录**：`curriculum/Learn Python with AI IDE/Module_X_.../`

### Stanford CS146S (配套演练)
- **架构**：**Week 制**（跟随 Stanford 进度）。
- **内容**：基于斯坦福 CS146S 的**配套讲义**。
- **目标**：应用基础技能，构建 Agent、RAG 等实际 AI 应用。
- **目录**：`curriculum/Stanford CS146S/weekX/`

## 3. 课程结构标准 (Lesson Structure)
每节课名为 `M{Module}_{Sequence}_{Topic}.md`（如 `M1_2_Terminal_Basics.md`）。

### 3.1 标准页头 (Standard Header)
每节课必须以统一的页头开始：
1.  **一级标题**：`# Module {X} Lesson {Y}: {Title}`
2.  **引用框 (Metadata Box)**：
    *   `> **课程体系：Learn Python with AI IDE**`
    *   `> **当前模块**：Module {X} - {ModuleName}`
    *   `> **本课目标**：描述本节课的核心交付物及技术点。`
3.  **分隔线**：元数据框后需紧跟 `---`。

### 3.2 核心内容结构 (7-Stage Structure)
课程内容建议参考以下逻辑链路进行组织，以确保知识点的平滑衔接与实践的有效落地。这套结构仅作为设计参考，不要求机械性照搬，可根据课题的具体需求进行灵活调整：

1.  **🔙 Backtrack (前修回顾)**：回顾上一课的核心概念，建立知识链条。
2.  **📅 Today's Teaser (今日预告)**：打破尴尬感，点出本课对后续学习（如 Python I/O、AI 协作）的深远意义，建立学习动机。
3.  **🏗️ Core Meaning (核心逻辑/机制)**：使用通俗易懂的比喻拆解底层机制，帮助理解。
4.  **🛠️ Practical Practice (操作与练习)**：包含具体的命令、代码操作及验证环节，培养实操习惯。
5.  **🔥 Efficiency Tips (提亮/进阶)**：提供提升效率的小技巧或高阶视角，展现 IDE 的灵活性（如终端布局、快捷键应用）。
6.  **✅ Checkpoint (自测与验收)**：明确的验收标准 (Pass Criteria)，让学员确认自己是否掌握。
7.  **🔄 Summary & Next Stop (复盘与预告)**：总结今日复盘思考，并丝滑地引出下一课内容。

## 4. 映射与维护 (Mapping)
- **核心映射**：基础课程的 Module 知识点需映射参考 **Andrew Ng** 的 Python 课程 (Module 1 & 2)。
- **文档维护**：
    - `docs/MAPPING.md`: 维护基础课程与 Ng 课程的知识点对照。
    - `docs/MODULE_TASKS.md`: 追踪 Module 建设进度。

## 5. 术语规范 (Terminology)
- 引用斯坦福课程时，使用 **CS146S**。
- **两岸术语差异 (Cross-Strait Terminology)**：在编写 `_TW.md` 版本时，必须遵循该地区的习惯，严禁直接繁简转换。
    - **SC (简体)**: 资源管理器, 扩展/插件, 终端, 命令面板, 标签页, 变量, 文件夹。
    - **TC (繁体)**: 檔案總管, 延伸模組/外掛, 終端機, 指令面板, 分頁, 變數, 資料夾。

## 6. 完成标准 (Definition of Done)
- 风格检查：无低幼隐喻，逻辑严密。
- 闭环验证：所有命令/代码在无前置假设的环境下可运行。
- 知识对齐：覆盖 Ng 课程对应的核心技术点。
## 7. 技术标准 (Technical Standards)
- **默认 Python 版本**：3.13 (现代、稳定、高性能)。
- **默认虚拟环境名**：`ai_course`。
- **环境管理工具**：Miniconda (建议开启 PATH)。
## 8. 课程埋线与回调 (Planned Transitions)
- **M1L4 -> M1L6**: 在 M1L6 (IDE) 中必须回调 M1L4 的虚拟环境概念。
    - **隐喻**：將虚拟环境比作“电源插座”，将 IDE 里的解释器设置比作“插头”。
    - **动作**：手动将 IDE “插上” M1L4 创建好的 `ai_course` 插座。

## 9. Git 协作准则 (Git Collaboration)
- **绝对禁令 (Absolute Prohibition)**：严禁在未获得用户明确指示或明确授权的情况下，执行任何 Git 操作（包括但不限于 `git add`, `git commit`, `git push`, `git branch` 等）。
- **确认流程**：作为 AI Agent，即便完成了代码修改且自认为逻辑闭环，也**必须**等待用户确认修改内容后。仅在用户明确下达 Git 相关指令（如“请提交”、“Push 吧”、“可以 Git 了”）后，方可启动 Git 提交流程。
