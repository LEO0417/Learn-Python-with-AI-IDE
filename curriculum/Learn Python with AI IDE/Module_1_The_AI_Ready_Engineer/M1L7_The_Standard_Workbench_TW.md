## 1. 工程邏輯：工作台的秩序與效率

### 1.1 什麼是專案？(The Project)
在程式設計中，**專案 (Project)** 包含了一次開發任務所需的所有程式碼、數據和設定檔案。
*   **初級視角**：專案等同於一個普通的資料夾。
*   **專業視角**：專案是你的「工程基座」。你需要清晰地劃定：
    *   **邏輯 (Scripts)**：你寫的程式碼檔案。
    *   **原料 (Data)**：專案執行需要的數據。
    *   **說明書 (README)**：專案的「身份證」，告訴別人（以及 AI）這個專案是做什麼的。

### 1.2 為什麼必須規範化？
1.  **可移植性 (Portability)**：規範的結構能確保你的程式碼在分享給他人或換台電腦時，依然能直接執行。
2.  **更有效地與 AI 協作**：理解根目錄、子目錄、相對路徑，能讓你更清晰地向 AI 表達你的需求，讓協作更精準。

## 2. 理解映射：IDE 檔案總管與物理磁碟

IDE 左側的**檔案總管 (Explorer)** 並不是一個獨立的虛擬空間，它是**本地磁碟資料夾在 IDE 中的「即時鏡像」**。

### 2.1 為什麼它如此重要？
檔案總管是你工程資產的「物理真相」。
*   **對於你**：它是程式碼、數據、文件的棲息地。
*   **對於 AI**：這是關鍵點——**AI 的「大腦」也是檔案**。在 Antigravity 中，AI 的長期記憶（Skills）和工作流（Workflows）本質上都是存儲在 `.agent` 等特定資料夾下的 `.md` 檔案。
*   **結論**：掌握了檔案總管，你就掌握了查看和修改 AI 認知的權限。

### 2.2 即時同步邏輯
你可以進行以下驗證：
1.  在 **Explorer (檔案總管)** 側邊欄的檔案列表下方空白處點擊右鍵，選擇「新建資料夾」，命名為 `my_demo`。
2.  打開你的物理磁碟窗口，你會發現資料夾瞬間同步生成了。

**💡 效率貼士**：忘記側邊欄在哪了？試試 `Cmd + B` (Mac) 或 `Ctrl + B` (Windows) 來快速切換顯示它。

## 3. 核心概念：絕對路徑 vs 相對路徑

這是區分「業餘玩家」與「專業工程師」的關鍵認知。

### 3.1 座標 vs 門牌號
*   **絕對路徑 (Absolute Path)**：就像地球上的「經緯度」。
    *   例如：`C:\Users\Leo\Desktop\project\data.csv`。
    *   **局限性**：如果你把專案發給朋友，他電腦上可能沒有相同的資料夾結構（比如使用者名稱不同），程式碼就會報錯。
*   **相對路徑 (Relative Path)**：就像是「在這個大樓的二樓左側」。
    *   例如：`./data.csv`。
    *   **優勢**：無論這棟大樓整體搬遷到哪裡（拷貝到哪個槽或哪台電腦），只要你在專案裡，總能透過相對位置找到檔案。

**建議**：**盡量在程式碼中使用相對路徑**。這樣編寫的程式碼更靈活，也更容易分享。

## 4. 重點實驗：重構並驗證「工作台」

### 4.1 搬遷資產
在 IDE 的檔案總管中：
1.  **創建目錄**：新建資料夾 `scripts` 和 `data`。
2.  **創建說明書**：在根目錄下新建一個檔案 `README.md`。暫時留空即可，它是我們下一課與 AI 協作的核心靶場。
3.  **移動資產**：將 `ide_demo.py` 以及之前的腳本全部拖入 `scripts` 中。

### 4.2 程式碼驗證：我是誰？他在哪？
1.  在 `scripts/` 下新建一個檔案 `path_check.py`，寫入：
    ```python
    import os

    print("--- 路徑分析儀表盤 ---")
    
    # 1. 我在哪？(The Anchor)
    # 打印當前的工作台錨點 (Current Working Directory)
    cwd = os.getcwd()
    print(f"當前工作台錨點 (CWD): {cwd}")

    # 2. 相對路徑 vs 絕對路徑演示
    # 假設我們要找 data 資料夾
    relative_path = "./data" 
    absolute_path = os.path.abspath(relative_path)

    print(f"相對路徑 (我們寫的): {relative_path}")
    print(f"絕對路徑 (系統算的): {absolute_path}")
    ```

> **💡 程式碼小科普：**
> - **`import os`**：就像是叫來一名「專業向導」（Operating System），它專門負責指揮 Python 與作業系統打交道。
> - **`os.getcwd()`**：翻譯過來就是：「向導，請告訴我，我現在所在的這個工作台錨點在哪？」
> 
> *註：關於它們的底層語法細節，我們在 **Module 2** 深入攻克 Python 核心概念時會詳細拆解，現在你學要把它當成一個「位置探測器」即可。*

2.  **執行並觀察**：你會發現，IDE 預設將你**開啟資料夾**的那一層級識別為整個工程的錨點。

## 5. 進階：由靜轉動——管理你的動態儀表盤 (Terminal Tabs)

剛才我們處理的是「靜態」的檔案結構，但在實際開發中，AI 常常會化身為「自動駕駛員」，在終端機裡為你設定環境、執行程式碼、修復 Bug。如果你不熟悉多終端機的操作，就很容易在 AI 同時開啟多個任務流時感到「一頭霧水」。這就需要用到我們的第二大支柱——**終端機 (Terminal)**。

一個高效的工程師不會只有一個儀表盤。在真正的開發場景中，我們需要**「並行操作」**。

### 5.1 實戰案例：靈活的「雙儀表盤」模式

專業的工程師絕不會被單一的視窗限制死。特別是在 AI 代勞的時代，你經常會遇到這種場景：AI 正在主終端機執行一個耗時很久的任務，但你又想在旁邊開個新視窗檢查結果。

這就需要一種通用的**「執行與核驗分離」**佈局作為參考：

1.  **場景模擬 A (執行區)**：在一個視窗專門負責執行程式碼（例如 `python scripts/path_check.py`）。這個視窗主要用來觀察程式碼吐出來的即時結果。
2.  **場景模擬 B (核驗區)**：利用 **Split (分屏)** 或新建分頁開啟第二個視窗。當程式碼正在執行時，你可以在這裡從容地輸入 `conda info` 檢查環境，或者用 `ls -R` 確認檔案搬遷是否成功，而不需要中斷第一個視窗的輸出。

**💡 儀表盤哲學 (Dashboard Philosophy)：**
這就像駕駛賽車：**「擋風玻璃」**看路況，**「儀表盤」**看數據。這種分工讓你能實現視線在「執行結果」與「系統狀態」之間的秒級切換，保持「感知不中斷」。

**🔥 高級進階技巧：將終端機「推向重心」**
如果你正在處理一個非常複雜的邏輯，終端機的輸出可能比程式碼本身更重要。這時，你可以**用滑鼠按住終端機的標籤欄，將其直接拖拽到螢幕右上方的編輯器區域**。
*   這樣，終端機就會像一個程式碼檔案一樣佔據視覺中心，讓你擁有更大的視野來監控那些滾滾而來的執行數據。
*   **記住**：佈局是活的，根據你當前的「戰鬥重點」隨時調整視角，才是真正的工程化思維。
